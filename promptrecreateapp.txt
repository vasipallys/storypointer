Last updated: 2026-07-06

Build a complete, production-ready application named “Story Pointer”. Do not create a mockup or partial prototype. Implement the backend, frontend, persistence, streaming workflow, tests, setup files, sample-data scripts, README usage flow, and this recreation prompt.

PRODUCT PURPOSE

Story Pointer is an evidence-led story-point estimator and project-planning workspace for React/Spring teams. It combines:

1. A checkpointed AI estimation pipeline that produces modified-Fibonacci points: 1, 2, 3, 5, 8, or 13.
2. A persistent interactive C4 workspace:
   - L1 = system / initiative
   - L2 = container / epic
   - L3 = component / story
   - L4 = code / task
3. An L1 operating-plan workspace for detailed requirements, reviews/audit, Word/PowerPoint exports, tribes/squads, people, delivery plans, cost, architecture, and infrastructure.
4. Jira, repository-scan, and spreadsheet ingestion.
5. Deterministic point roll-ups from stories to epics and initiatives.

CORE ESTIMATION PIPELINE

Implement this checkpointed LangGraph flow:

score_parameters
→ identify_drivers
→ compare_to_anchors
→ derive_points
→ conditional spike_split_branch when points == 13 or uncertainty is High
→ write_plain_language_reasoning
→ detect_hidden_tasks
→ assess_risks
→ recommend_split

Requirements:

- Define EstimationState and one Pydantic structured-output schema per node.
- Score complexity, volume, uncertainty, React scope, Spring scope, existing-code scope, dependencies, NFRs, testing, compliance/audit, familiarity, and Definition-of-Done overhead.
- Identify the two or three decisive drivers and compare with six fixed stories in backend/anchors.py.
- Return only 1/2/3/5/8/13.
- Every final result must contain both tldr and plain_language_why.
- Never emit points before those explanations exist.
- Include layer effort, person-day range, risks, assumptions, hidden tasks, spike advice, and split recommendations.
- A 13 always recommends splitting and supplies proposed stories.
- Re-estimation with the same session_id must reuse the prior checkpoint.
- Parse structured JSON defensively across provider envelopes, markdown fences, list echoes, and parsed/raw variants.

LLM BOUNDARY

- Provider/model strings must not appear outside backend/llm/factory.py.
- Native providers use LangChain init_chat_model.
- OpenAI-compatible providers share ChatOpenAI with base_url.
- Groq uses JSON-mode structured output.
- “mock” requires no API key and returns deterministic structured data based on a stable title hash.
- API-key validation is provider-specific and belongs in validate_factory_config.
- Startup configuration errors must not crash FastAPI. Report them from /health and return 503 from estimation routes.

C4 DOMAIN AND UX

- Persist projects, repository links, Jira links, C4 elements, relations, and artifact links in SQLite.
- A child’s parent must be exactly one level above.
- L1 maps to initiative, L2 to epic, L3 to story, and L4 to task.
- Bugs may tag L3/L4. Tech debt and architecture flow may tag L2/L3.
- Only L3 and L4 are directly estimable.
- L1/L2 totals are deterministic sums, never LLM output.
- Proposed elements do not enter roll-ups.
- Element estimation injects parent chain, relations, technology, and code path as c4_context.
- Hidden tasks seed proposed L4 children; split recommendations seed proposed sibling L3 stories.
- Imports and replays are idempotent.
- Use a React Flow canvas with persistent positions and labeled relations.
- Click any L1-L4 node to select it and keep the C4 canvas open.
- For L1, show an operating-plan summary inside the existing inspector sidebar, alongside description editing and Jira actions.
- The L1 summary shows squads, people, monthly run-rate, approved budget, diagram count, work-package count, at-risk work, and allocated FTE.
- Provide a “More details” button in the L1 summary that opens the full plan with that exact initiative selected.
- Double-click L1-L3 to drill; breadcrumbs navigate back.
